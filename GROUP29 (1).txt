1. Introduction 
This document presents the Requirements Analysis Report for the EV Charging Station Network 
Management System, developed for the Fundamentals of Software Engineering course at Ege 
University. The primary objective of this project is to define and structure the software requirements 
necessary to build a comprehensive, reliable, and scalable solution for managing electric vehicle 
charging infrastructures. 
The proposed system aims to seamlessly connect various stakeholders, primarily serving EV drivers, 
station operators, and system administrators. As the adoption of electric vehicles grows, drivers face 
challenges such as finding available stations, ensuring charger compatibility, and managing payments. 
To address these, our system introduces a robust set of core functionalities, including an interactive 
station catalog with real-time availability, automated vehicle-to-charger compatibility verification, and 
a conflict-free reservation workflow. Additionally, the platform ensures secure financial transactions 
by processing active charging sessions and automatically deducting fees based on exact energy 
consumption. 
To deliver an optimal and context-aware user experience, the system is conceptualized around a 
dual-platform architectural strategy. EV drivers will interact primarily through a mobile application, 
which leverages the Google Maps API to provide location-based services, real-time navigation, and 
on-the-go session monitoring. Conversely, a comprehensive web-based dashboard will be developed 
for system administrators and station operators, empowering them to oversee hardware health, 
manage administrative configurations, and extract analytical reports. 
The requirements detailed in the following sections have been systematically elicited, classified, and 
negotiated in accordance with established software engineering principles. By utilizing 
Viewpoint-Oriented Requirements Definition (VORD) to capture diverse stakeholder needs and the 
MoSCoW method for strict feature prioritization, this report establishes a highly structured, legally 
compliant, and technically feasible foundation for the subsequent architectural design and 
implementation phases.

2. Identification of Viewpoints 
Principal Viewpoints of the System 
To ensure a comprehensive understanding of the system's requirements, we have identified and 
categorized the stakeholders and interacting entities into three main viewpoint classifications: 
Interactor (Direct), Indirect, and Domain. 
●  Interactor Viewpoints:  EV Driver, Station Operator,  System Administrator, Payment Gateway, 
Energy Provider, Google Maps API. 
●  Indirect Viewpoints:  Marketing Department, Security Department, Legal/Regulatory 
Compliance.
●  Domain Viewpoints:  Industry Standards (OCPP), Government Regulations, Grid 
Management Standards. 
Description of each Viewpoint 
Below is a detailed description of each identified viewpoint, outlining their specific roles, interactions 
with the system, and their primary areas of concern. 
Interactor Viewpoints (Direct Actors and External Systems) 
●  EV Driver:  The primary end-user of the system. They  are responsible for registering their 
electric vehicles, searching for available stations on the map, making time-specific 
reservations, and monitoring their active charging sessions. Their main concern is usability, 
real-time accuracy of station availability, and seamless payment processing. 
●  Station Operator:  The personnel responsible for the  physical oversight and maintenance of 
the charging hardware. They interact with the system to mark faulty chargers as "out of 
service," which automatically triggers the cancellation of affected reservations to maintain 
network reliability. 
●  System Administrator:  The overarching manager of the  platform. They are responsible for 
maintaining the database of charging stations, defining pricing models, updating operating 
hours, and extracting analytical reports regarding revenue and station utilization. 
●  Payment Gateway:  An external financial service that  securely handles user wallet top-ups. It 
automatically deducts charging fees based on energy consumed and processes refunds for 
canceled reservations according to system rules. 
●  Google Maps API:  An external service integrated into  the system to provide geographical 
context. It supplies interactive map data, displays real-time station markers (available, 
occupied, offline), calculates distances, and provides navigation routes for the drivers. 
●  Energy Provider:  The external utility entity supplying  the electricity. The system must 
interact with this provider to periodically report total energy consumption to help balance 
the grid load and facilitate dynamic "Smart Charging" speeds during peak hours. 
Indirect Viewpoints (Internal Stakeholders) 
●  Marketing Department:  An internal business unit that  does not directly interact with the 
core charging workflow but relies on the system's data. They analyze users' favorite stations, 
usage habits, and active user numbers to create personalized offers and summarize user 
behavior. 
●  Security Department:  The entity focused on the platform's  digital safety. They are concerned 
with enforcing access restrictions, managing secure API keys, and ensuring that user data 
and financial transactions are protected against breaches. 
●  Legal / Regulatory Compliance:  An oversight viewpoint  ensuring the system operates within 
the bounds of the law. They require the system to retain charging history data for legally 
mandated financial reporting periods and mandate that user location and vehicle data are 
stored securely according to personal data protection standards (e.g., GDPR). 
Domain Viewpoints (Constraints and Standards) 
●  Industry Standards (OCPP):  Represents the technical  communication protocols and 
hardware constraints. This viewpoint dictates how the software must communicate with the 
physical charging stations to ensure interoperability. 
●  Government Regulations:  Represents the legal and safety  boundaries imposed by state 
authorities. This includes compliance with safety certifications and fire protection standards 
required for the physical operation of charging stations. 
●  Grid Management Standards:  Represents the overarching  rules for power distribution. This 
viewpoint enforces constraints on the system to prevent grid overloads, directly influencing 
features like dynamic charging limits and peak hour energy management.

  7. Requirements Prioritization and Negotiation 
 7.1 Requirements Prioritization using MoSCoW Method 
 To effectively manage development resources and ensure the core functionalities of the EV Charging 
 Station Network Management System are delivered successfully, we utilized the MoSCoW 
 prioritization technique. 
 ●  Must Have (Critical for system operation):  These are non-negotiable requirements. Without 
 them, the system cannot function as an EV charging network. 
 ○  Station Catalog & Real-time availability status. 
 ○  Vehicle Registration and compatibility checking. 
 ○  Core Reservation workflow and conflict handling. 
 ○  Charging Session processing and secure payment collection. 
 ●  Should Have (Important but not immediately life-threatening to the launch):  High-priority 
 features that significantly improve user experience and administration. 
 ○  Issue reporting and automatic cancellation of affected reservations. 
 ○  Administrative reporting (revenue, utilization). 
 ○  System notifications via Email/SMS for reservation updates.

 ●  Could Have (Desirable but expendable):  Enhancements that can be delayed to later versions 
if time or budget constraints arise. 
○  Marketing department analytics for personalized offers. 
○  Tiered loyalty programs or digital waitlists for fully booked stations. 
●  Won't Have (Out of scope for this release):  Features  agreed to be excluded from the current 
2025-2026 project scope. 
○  Vehicle-to-Grid (V2G) energy discharging functionalities. 
○  Physical hardware manufacturing of the charging stations. 
7.2 Requirements Negotiation and Conflict Resolution 
During the requirements elicitation phase, overlapping or conflicting demands from different 
viewpoints were identified. Through stakeholder negotiation, these conflicts were resolved to balance 
user satisfaction, business goals, and technical/legal constraints. 
●  Conflict 1: Data Utilization vs. Privacy 
○  Stakeholders:  Marketing Dept. vs. Legal/Government Reg. 
○  The Issue:  The Marketing Department requested deep  access to user location 
histories and habits to create personalized campaigns. However, Government 
Regulations strictly mandate that location data must be protected under GDPR 
standards. 
○  Resolution:  A compromise was reached. The system will use real-time location 
strictly for navigation and distance calculation. Marketing data will be fully 
anonymized, and users must explicitly "opt-in" to receive personalized offers, 
satisfying both business needs and legal compliance. 
●  Conflict 2: Maximum Performance vs. Grid Stability 
○  Stakeholders:  EV Driver vs. Energy Providers. 
○  The Issue:  EV Drivers expect maximum power output (e.g., full 150kW) at all times 
for the fastest charging experience. Conversely, Energy Providers require load 
balancing during peak grid hours to prevent local power outages. 
○  Resolution:  The implementation of "Smart Charging".  The system will prioritize the 
EV driver's reservation but retains the domain-level authority to dynamically reduce 
charging speeds if a critical demand signal is received from the Energy Provider. The 
user will be notified of the adjusted estimated completion time via the app. 
●  Conflict 3: Booking Flexibility vs. Station Utilization 
○  Stakeholders:  EV Driver vs. System Administrator. 
○  The Issue:  Drivers want the ability to book stations days in advance and hold them 
indefinitely. Administrators want to maximize station turnover and revenue. 
○  Resolution:  A strict booking regime was negotiated.  Users cannot reserve more than 
24 hours in advance, and sessions are capped at a maximum of 2 hours. To 
compensate the drivers for this strictness, an automated refund policy was 
established for operator-side cancellations.

9. Use Cases of the Main Scenarios 
Use Case 1: Create Reservation 
This is the most critical workflow of the system, encompassing the management of vehicle 
compatibility and time conflict resolution. 
●  Inputs:  Station selected via the map, a specific date and time slot, and pre-verified registered 
vehicle details. 
●  Actors:  EV Driver. 
●  Action:  The user selects a station on the interactive map and reserves a charging slot for a 
specific date and time. During this process, the system verifies the hardware compatibility 
between the user's vehicle and the selected charger. The system enforces business rules 
(e.g., maximum duration) and prevents double-booking. 
●  Outputs:  Turn-by-turn navigation to the selected charger  provided via Google Maps 
integration, and a reservation confirmation notification sent to the user. 
●  Exceptions:  Hardware incompatibility between the vehicle and the charger. Time slot conflict 
(double-booking attempt). Attempting to reserve a session exceeding the 2-hour maximum 
limit or booking more than 24 hours in advance. The selected station currently being in an 
"offline" state. 
Use Case 2: Process Charging Session 
This is the core operational process where financial transactions and station-to-system 
communications occur. 
●  Inputs:  Consumed energy data (in kWh) transmitted  from the charging unit. 
●  Actors:  EV Driver, Payment Gateway. 
●  Action:  The system records the active charging session. Upon completion, the session is 
processed to calculate the final cost. Charging fees are automatically deducted from the 
user's wallet based on the exact amount of kWh consumed. 
●  Outputs:  Real-time progress display showing current  kWh, estimated time remaining, and 
accrued cost so far. Generation of a digital receipt. Notifications indicating the start and 
completion of the charging session. An updated charging history record containing session 
costs. 
●  Exceptions:  The user having an insufficient (low) wallet balance to initiate or cover the 
session. 
Use Case 3: Vehicle Registration & Compatibility Setup 
This is a mandatory workflow that enables user-based personalization of the system and forms the 
foundation for pre-reservation hardware compatibility checks. 
●  Inputs:  Vehicle Brand, Model, Battery Capacity (kWh),  Connector Type (Type 2, CCS, 
CHAdeMO), and License Plate Number.

●  Actors:  EV Driver. 
●  Action:  The user logs into the system and inputs the  technical specifications and license 
plate information of their electric vehicle. The system queries the database to verify if the 
license plate number is unique. The system then saves the connector and battery details to 
the user's profile to act as a compatibility filter for future reservations. 
●  Outputs:  A UI notification confirming the successful  addition of the vehicle. An updated and 
linked user-vehicle profile record in the database. 
●  Exceptions:  The entered license plate number is already registered to another account in the 
system (unique key constraint violation). The submitted technical details are incomplete or in 
an unrecognized format.
